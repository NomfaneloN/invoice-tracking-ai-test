## Root Cause Analysis

The failing endpoint `SetPaymentApprover` is called as part of the post-Certify workflow action. It should resolve the next approver (the Business Unit Supervisor) for the selected Business Unit. Based on the error pattern, **the backend cannot resolve a supervisor for Tania Smith's Business Unit**, so the API throws 500.

Prior memory notes match:  
> "LOGIS TC-01 blocker — supplierId=undefined bug fixed by dev on 2026-04-20; TC-01/02 passed; TC-03/04 blocked by routing mismatch for ORDER-00024"

The supplierId fix from 2026-04-20 let TC-01/TC-02 pass on that day, but the downstream SetPaymentApprover routing for ORDER-00024 appears to still be broken today. Three other recent LOGIS records on ORDER-00024 are also stuck at "Certified" status (PAY5470/2026 — 20/04, PAY5103/2026 — 14/04) which is consistent with the same bug.

## Recommendations

1. **Raise backend bug for `SetPaymentApprover` returning 500** on LOGIS Certify Invoice submits. Investigate whether the Business Unit → Supervisor mapping for Tania Smith's unit is missing a record, or the API lookup throws when the supervisor isn't set.
2. **Surface 500 errors to the user.** Silent failure is a worst-case UX — users will re-click Submit assuming their first click didn't land.
3. **Retry LOGIS test** once the SetPaymentApprover endpoint is fixed. The test plan from TC-03 onwards is ready to be exercised.
4. **Workaround for test (if immediately needed):** create or reuse an Order whose Business Unit has a resolved supervisor (e.g. one of the older orders that completed TC-02 historically). The test plan did not specify an alternate order — dev input needed.
