3. Admin Payment Stubs Import — successfully uploaded `training STUB E2E-20260415.txt`; Is Success: Yes, Payments Confirmed: 1; BUT workflow step did not advance (PAY5154 remained at "Final Authorise Payment" in both Thulilem's and Admin's inbox)

5. Direct API call attempts (SubmitUserAction, ProcessUserAction, Complete, ExecuteAction) — all returned 404

**Post-stub-import state:** After the stub import confirmed 1 payment for ITS-LOGIS-001, the `activeTodoItems` became empty for both todo holders. The workflow instance is now in a state where the "Final Authorise Payment" step shows as assigned and active but no user can take action on it.

## Recommendations

1. **Fix "Final Authorise Payment" form — Critical:** The action sub-form (payment stub upload, confirmation checkbox, Submit) does not render. Investigate why `userAuthorizedToAction` returns `false` for assigned users. Also verify that the Payment Stubs Import does not invalidate the pending todo items for this step before it is manually submitted.

2. **Fix inline grid save/close bug in Invoices table — High (TC-06):** Clicking save-circle or close-circle in the Payment Number cell creates a new invoice row instead of committing/discarding the edit. This corrupts the Total Amount and blocks Submit. The row's React state event handlers appear to fire a create operation instead of an update.

3. **Fix Order entity picker Enter key search — Medium (TC-01):** Pressing Enter in the order search box returns "No Data". Only the search icon click works. Standard keyboard input should trigger the search.

4. **Investigate routing deviations — Medium:** Multiple steps were routed to unexpected users (Thulilem for TC-03, TaniaSmith for TC-06/TC-07). TC-09 was skipped entirely. Review the workflow routing rules for LOGIS Request For Payment to confirm expected role assignments per step.

5. **Fix "Live" mode badge — Low:** Mode badge consistently shows "Live" when localStorage mode is "latest". Should reflect actual configured mode.
