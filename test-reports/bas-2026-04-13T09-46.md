- Root cause: re-using ITS-E2E-001 across multiple test runs creates duplicate SOURCE DOC NUMBER entries; the import selects the wrong record when multiple matches exist, regardless of file format
- Defect: system does not filter matches by workflow state — it should only consider records currently at the "Upload Captured Invoices Report From BAS" step when authorising a payment

**Defect recommendation:** The BAS import should match on SOURCE DOC NUMBER AND workflow state (record must be at "Upload Captured Invoices Report From BAS" step). Using a unique invoice number per test run (e.g. ITS-E2E-20260413) would avoid the ambiguity in future runs.

## Recommendations

1. **Defect (High):** Fix BAS import matching logic to resolve on workflow instance state, not just SOURCE DOC NUMBER. When multiple records share an invoice number, the import should target the one currently awaiting the "Upload Captured Invoices Report From BAS" step. Alternatively, reject the import with a clear error message identifying the ambiguity. Confirmed across 4 import attempts with two different file formats.

2. **Test Data (Medium):** Use unique invoice numbers per test run (e.g., `ITS-E2E-[YYYYMMDD]`) to prevent SOURCE DOC NUMBER conflicts across runs. Update TC-01 test data accordingly. PAY4848/2026 (invoice ITS-E2E-20260409) in My Items confirms this approach works.

3. **UI/UX (Low):** Resolve the mode badge display bug ("Live" shown when mode is "Latest"). Misleads testers who check the badge to confirm mode.

4. **UI/UX (Medium):** Add feedback to BAS import result when Payments Authorised = 0 — surface a warning listing which invoice numbers were found, which were matched, and why authorisation was not triggered.

5. **Workflow Completion:** TC-09 and TC-10 remain untested in this run. Re-run from TC-01 with a unique invoice number (e.g. ITS-E2E-20260413) to reach the Attach Payment Stub and Capture Filing steps.
